## Pillar C1 — Managed Infra & KMS

**Objective:** Move off the single-box docker-compose dev stack to managed AWS services, and lift the Phase B local-AES PII key into a real KMS-backed key provider so a host compromise or DB dump alone cannot expose victim PII.

### C1.1 — KMS-Backed Key Provider

- **Location:** create `backend/app/services/security/kms_provider.py`; modify `backend/app/services/security/pii_vault.py`; add config to `backend/app/config.py`.
- **Action:** Phase B's `pii_vault.py` uses a local `pii_encryption_key` for AES-256-GCM. **Introduce a `KeyProvider` interface** (`get_data_key() -> bytes`, `decrypt_data_key(wrapped) -> bytes`) with two implementations: `LocalKeyProvider` (the current behavior — dev only) and `KMSKeyProvider` (envelope encryption via AWS KMS: the master key lives in KMS; a per-tenant/per-table data encryption key (DEK) is generated by KMS, wrapped, and stored alongside ciphertext). `pii_vault` resolves the provider from `settings.kms_key_id` — when set, use KMS; when empty, fall back to local (dev). The migration path: existing Phase B ciphertext was encrypted with the local key directly (not envelope-wrapped), so add a `key_version` byte prefix to new ciphertext and a migration job to re-encrypt legacy rows on first decrypt (lazy re-encryption) — never a big-bang re-encrypt that locks the table.
- **Config:** `kms_key_id: str = ""`, `kms_region: str = "ap-south-1"`, `aws_access_key_id: str = ""`, `aws_secret_access_key: str = ""` (empty = use IAM role in prod, static creds only for local E2E). Fail-fast in non-dev if `pii_encryption_key` is empty AND `kms_key_id` is empty (at least one must be set).
- **Verification:** Unit test: `LocalKeyProvider` round-trips (regression of Phase B behavior); `KMSKeyProvider` round-trips against a `moto` (mocked KMS) fixture; the lazy re-encryption path upgrades a legacy ciphertext to envelope-wrapped on first read and returns the plaintext.

### C1.2 — Managed Postgres & Redis Connection Hardening

- **Location:** modify `backend/app/config.py`, `backend/app/database.py`, `backend/app/main.py` lifespan; update `infra/docker-compose.yml` (dev parity), `infra/helm/trustshield/values.yaml`.
- **Action:** (1) Add a **connection pool sizing** config (`db_pool_size: int = 10`, `db_max_overflow: int = 20`, `db_pool_timeout: int = 30`, `db_pool_recycle: int = 1800`) and wire into the `create_async_engine` call in `database.py`. (2) Add `pgbouncer` as the pooler in front of RDS in prod (the `infra/pgbouncer.ini` stub already exists — complete it with `pool_mode = transaction`, `max_client_conn = 1000`, `default_pool_size = 25`). (3) Add an SSL requirement: when `environment != development`, the DB URL must use `sslmode=require` and Redis must use `rediss://` (TLS). Validate this in the lifespan startup check and refuse to boot otherwise. (4) Document the managed-RDS connection params in `infra/DEPLOYMENT.md`.
- **Verification:** Unit test that the engine is built with the configured pool size (mock `create_async_engine` and assert kwargs); startup check test that a non-dev config with a non-SSL DB URL raises `SystemExit(1)`; dev docker-compose still boots unchanged.

### C1.3 — Secrets Management (AWS Secrets Manager)

- **Location:** create `backend/app/services/security/secrets_loader.py`; modify `backend/app/config.py` to optionally hydrate from Secrets Manager.
- **Action:** Phase B reads all secrets from env vars / `.env`. For prod, add an optional `secrets_manager_prefix: str = ""` config; when set, a `SecretsLoader` fetches a named secret (e.g., `trustshield/prod/db`, `trustshield/prod/stripe`, `trustshield/prod/pii_key`) from AWS Secrets Manager during startup and injects the values into `os.environ` *before* `Settings()` is instantiated. This keeps secrets out of the container environment / Helm values (which are often visible in kubectl describe). Keep env-var loading as the dev fallback. Cache the fetched secrets in-process (they don't change at runtime).
- **Verification:** Unit test with `moto` that `SecretsLoader` populates `os.environ` from a mocked secret; dev boot (no prefix) is unchanged; a test that secrets already in env are not overwritten by Secrets Manager (env wins for explicit overrides).

### C1.4 — IaC: Terraform for Managed Resources

- **Location:** create `infra/terraform/` with modules for `rds`, `elasticache`, `msk`, `s3` (ML artifacts + backups), `kms`, `secrets`, `ecr`; a root `main.tf` and per-env `prod.tfvars` / `staging.tfvars`.
- **Action:** Codify the managed resources so infra is reproducible and auditable (Phase A/B infra was docker-compose + a Helm chart that assumed managed services existed). Modules: (1) `rds` — Postgres 16, encrypted-at-rest with the KMS key, multi-AZ, automated backups 30-day retention, parameter group with `log_statement = ddl`. (2) `elasticache` — Redis 7, encryption-in-transit + at-rest, auth token. (3) `msk` — Kafka (or skip if `event_backend=redis` — keep it conditional). (4) `s3` — two buckets: `ml-artifacts` (versioned, for ONNX models + gold reports) and `backups` (versioned, object-lock for WORM compliance evidence). (5) `kms` — the master key referenced by C1.1 and RDS/ElastiCache/S3. (6) `ecr` — for the API/worker container images. Output all connection strings as Secrets Manager entries (C1.3).
- **Verification:** `terraform plan` for staging is clean and idempotent (a second `plan` shows no diff); `terraform validate` passes; the KMS key policy restricts decrypt to the API and worker IAM roles only.

## Pillar C2 — Job Scheduling & Workers

**Objective:** Replace the in-process `asyncio.create_task` background loops (Phase B's B1.8 nightly rollup, B2.5 drift worker, B3.1 audit verify) with a real scheduled Celery beat configuration, and verify the worker containers actually run them. Today the Celery app exists (`backend/app/workers/celery_app.py`) but no beat schedule is registered and no tasks are bound — the jobs would never fire.

### C2.1 — Celery Beat Schedule

- **Location:** modify `backend/app/workers/celery_app.py`; create `backend/app/workers/tasks/` package with `billing_tasks.py`, `ml_tasks.py`, `compliance_tasks.py`.
- **Action:** Register a `celery_beat_schedule` in `celery_app.py` with entries: `nightly-usage-rollup` (crontab `5 0 * * *` → `billing_tasks.nightly_usage_rollup`), `stripe-metering` (`30 0 * * *` → `billing_tasks.submit_stripe_metering`), `usage-retention` (`0 3 * * 0` → `billing_tasks.purge_old_usage_events`), `drift-check` (`0 1 * * *` → `ml_tasks.run_drift_check`), `audit-verify-daily` (`15 1 * * *` → `compliance_tasks.verify_audit_chain_window`), `audit-verify-full` (`0 4 * * 0` → `compliance_tasks.verify_audit_chain_full`), `backup-audit` (`0 5 * * 1` → `compliance_tasks.verify_backups`). Each task is a thin Celery-wrapper around the existing service functions (`billing/jobs.py`, `ml/drift_worker.py`, `audit/verify_job.py`, `compliance/backup_audit.py`) — do NOT duplicate logic; import and call. Give each task `bind=True`, `max_retries=3` with exponential backoff, and a per-task timeout (`soft_time_limit`) so a stuck job can't block the worker.
- **Wiring:** The existing `infra/docker-compose.yml` has a `worker` service — confirm it runs `celery -A app.workers.celery_app worker` AND `celery -A app.workers.celery_app beat` (or a separate `scheduler` container — beat should run in exactly one place). Add the same to the Helm chart deployment.
- **Verification:** Unit test that the beat schedule contains the expected crontab entries; a test that each task calls its underlying service function (mock the DB session) and respects the retry policy on failure; a live check that `celery inspect active` lists the scheduled tasks.

### C2.2 — Task Idempotency & Dead-Letter

- **Location:** modify the task modules from C2.1; add config `celery_task_default_queue`, `celery_task_deadletter_queue`.
- **Action:** Every scheduled task must be **idempotent** (Beat can double-fire on restart). For the rollup/retention/verify jobs this is already true (they're reconciliation-style). Add a `task_id` based on `(task_name, scheduled_time_bucket)` and use Celery's `cache` backend or a Redis SET to dedupe within the window. Add a **dead-letter path**: after `max_retries`, publish to a `deadletter` queue and trigger an alert via the existing `alert_service` — a silently-failing nightly job is a compliance risk (e.g., audit verification not running for days). Surface dead-letter depth on the SRE dashboard (C4).
- **Verification:** Unit test that firing the same task twice in one window runs the underlying logic once; a test that exhausting retries enqueues to dead-letter and calls `alert_service`.

### C2.3 — Kafka Consumer Hardening (if event_backend=kafka)

- **Location:** modify `backend/app/workers/kafka_consumer.py`.
- **Action:** The consumer exists but is a stub. If `event_backend=kafka` in prod, harden it: consumer group per concern (`trustshield-analyze`, `trustshield-webhook`), at-least-once delivery with idempotent handlers (dedupe on `event_id`), manual commit only after successful processing, exponential backoff on deserialization errors (poison-pill handling → dead-letter topic, not infinite retry). If `event_backend=redis` (the default and simpler path), document that the Kafka consumer is not required and gate its startup on the config.
- **Verification:** Unit test with a mocked KafkaConsumer that a poison-pill message is moved to the dead-letter topic and the consumer continues; a test that a handler crash does not commit the offset (message is redelivered).

## Pillar C3 — Model Serving at Scale

**Objective:** Move the ONNX model out of the API process into a dedicated model server (Triton or BentoML) so the API can scale horizontally without each replica loading the multi-hundred-MB transformer, and so GPU/predict can be right-sized independently of the API tier. This was explicitly deferred from Phase B (B2.4 note).

### C3.1 — BentoML Model Service

- **Location:** create `backend/ml/serving/bentofile.yaml`, `backend/ml/serving/service.py`; add `bentoml>=1.2.0` to `backend/requirements.txt` (in a serving-only extras group if possible, to keep the API image lean).
- **Action:** Package the Phase B transformer + GBM ONNX artifacts (from `ml/artifacts/`) as a BentoML service exposing two RPCs: `classify_transformer(text) -> {label, confidence}` and `classify_gbm(features) -> {label, confidence}`. BentoML handles batching, GPU sharing, and autoscaling. The service loads the model once at startup (not per-request). Choose BentoML over Triton for Phase C because BentoML's packaging story (Bento → container → Yatai/Knative deploy) is simpler and the model is a standard transformer; Triton is justified only for high-throughput multi-model GPU serving (Phase D candidate).
- **Verification:** `bentoml serve` locally answers a classify RPC in <50ms with the transformer; a Bento is built (`bentoml build`) and containerized (`bentoml containerize`); the container starts and serves.

### C3.2 — Client Migration (API → Model Service)

- **Location:** modify `backend/app/services/nlp/model_loader.py`, `backend/app/services/nlp/classifier.py`; add config `model_service_url: str = ""`.
- **Action:** Add a `RemoteModelClient` alongside the existing in-process `OnnxModelLoader`. When `model_service_url` is set, `ModelLoader` returns the remote client; when empty (dev), the in-process loader. `classifier.py`'s tiered fallback (transformer → GBM → keywords) is unchanged — it just calls `loader.classify_transformer(text)` which now may be an HTTP/gRPC call. **Keep the in-process fallback:** if the model service is unreachable (timeout, 5xx), fall through to GBM then keywords exactly as the 50ms-timeout path does today (`model_fallback_total` counter increments). This means a model-service outage degrades gracefully, not hard-fails.
- **Config:** `model_service_url`, `model_service_timeout_ms: int = 100` (tighter than the in-process 50ms because of network). Fail-fast NOT required — an empty URL means in-process, which is valid for dev.
- **Verification:** Unit test that an empty `model_service_url` uses the in-process loader (regression); a test that a populated URL routes to the remote client and a remote timeout falls through to GBM; integration test that the classifier returns correct labels when pointed at a locally-served BentoML service.

### C3.3 — Model Artifact Store (S3)

- **Location:** modify `backend/ml/training/run_pipeline.py` (B2.1) to upload artifacts; create `backend/app/services/ml/artifact_store.py`.
- **Action:** The training pipeline (Phase B) writes artifacts to local `ml/artifacts/`. Add an S3 upload step: after a successful pipeline run, upload the artifact bundle (versioned by `model_version` + git SHA) to `s3://ml-artifacts/transformer/<version>/` and `s3://ml-artifacts/gbm/<version>/`, and write a manifest to the model registry table. The serving service (C3.1) and the in-process loader both resolve the active version from the registry and pull from S3 (with local disk cache). This decouples "train" from "serve" hosts and gives versioned rollback (`promote_model(previous_version)` just flips the registry pointer).
- **Verification:** Pipeline run with `--upload` pushes to a mocked S3 (moto); `artifact_store.resolve(version)` returns the S3 key; the serving service pulls and caches a version it didn't ship with.

## Pillar C4 — Observability & SRE

**Objective:** Turn the Prometheus counters (registered in Phase B's `main.py` lifespan) into actionable dashboards and alerts, and add the structured-logging/tracing needed to debug a multi-service system. Today metrics are emitted but there are no dashboards and alert routing is stubbed.

### C4.1 — Grafana Dashboards

- **Location:** create `infra/dashboards/` with `overview.json`, `billing.json`, `model.json`, `compliance.json`; reference them in `infra/helm/trustshield/templates/` (or document import).
- **Action:** Build four Grafana dashboards backed by the Prometheus metrics already emitted:
  1. **Overview** — request rate, p50/p95/p99 latency per route, error rate (5xx), DB pool utilization, Redis hit rate, active worker count.
  2. **Billing** — `billing_quota_denied_total` by plan, `stripe_webhook_total` by event type, usage-record lag (`billing_meter_lag_seconds`), MRR proxy (sum of active subscriptions by plan).
  3. **Model** — `model_fallback_total` by tier (the key SLA metric — keyword fallback should be ~0), request rate to the model service, drift PSI over time (from `drift_log`), shadow agreement rate, promotion history.
  4. **Compliance** — `audit_chain_break_total` (must be 0), nightly-job success/failure, backup-audit status, DPDP register last-reviewed age.
  Each dashboard templated by `environment` and `instance`. Use the `prometheus-client` metrics already defined; do NOT add new metric names without registering them in `main.py`.
- **Verification:** Dashboards import into Grafana without error (validate JSON schema); each panel has a non-empty query against an emitted metric; a reviewer can find "current model fallback rate" in <3 clicks.

### C4.2 — Alerting Rules & Routing

Route CRITICAL to PagerDuty/Slack `#trustshield-incidents`; WARNING to `#trustshield-ops`. Each alert's annotation links to the runbook in `infra/RUNBOOK.md`. The existing `alert_service` in `app/services/alerting/` should POST to the Alertmanager webhook (or directly to Slack) — unify on one path.

### C4.3 — Structured Logging & Trace Correlation

- **Location:** modify `backend/app/middleware/request_id.py`, the `RequestIDFormatter` in `main.py`; add OpenTelemetry spans in the hot paths (`analyze.py`, `classifier.py`, `pii_vault.py`).

## Pillar C6 — Scale Verification & Chaos

**Objective:** Prove the system meets the capacity and resilience targets before declaring production-ready. Phase B's load tests exist (`backend/tests/load/`) but were not run against the full managed stack. Phase C closes that loop.

### C6.1 — Load Profile & Capacity Ceiling

- **Location:** modify `backend/tests/load/test_analyze.py` and any locustfiles; create `infra/docker-compose.loadtest.yml` (already exists — extend it); document results in `infra/PERFORMANCE.md`.
- **Action:** Define the target load: 500 RPS sustained, 1500 RPS burst (30s), on `/analyze` with a representative payload mix (60% text chat, 30% webhook, 10% batch). Run the locust suite against a staging deploy that matches prod shape (C1 managed infra + C3 model service + C2 workers). Profile to find the bottleneck (likely DB pool, model-service latency, or GIL in the in-process fallback) and tune until p95 < 300ms on `/analyze` at 500 RPS and p95 < 800ms at burst. Capture: max RPS before SLO breach, latency at each tier, resource utilization (CPU/mem/DB connections/model-service GPU). Document the **per-replica capacity** so autoscaling HPA thresholds can be set.
- **Verification:** A passing load run at 500 RPS sustained for 10 minutes with p95 < 300ms and zero 5xx; `PERFORMANCE.md` records the numbers and the chosen HPA thresholds (`cpu > 70%`, `rps > 400/replica`).

### C6.3 — Chaos & DR Drill

- **Location:** create `infra/chaos/experiments.md` (documented manual + scripted chaos), `infra/DR_RUNBOOK.md` (disaster-recovery procedure).
- **Action:** Define and run chaos experiments, each with a hypothesis and pass criteria: (1) **kill an API replica** mid-traffic → no 5xx spike, load balancer drains within 5s; (2) **kill the model service** → `model_fallback_total{tier="keyword"}` rises, requests still return (degraded), alert fires; (3) **kill a worker** → scheduled jobs stall but recover on restart, no double-processing (C2.2 idempotency); (4) **partition Redis** → circuit breaker trips, write path degrades gracefully; (5) **DR drill** — restore the RDS snapshot to a fresh instance, point the app at it, verify the latest audit chain validates end-to-end (the real test of "can we recover compliance evidence after an incident"). Each chaos run is recorded (date, outcome) in `DR_RUNBOOK.md`.
- **Verification:** All four chaos experiments pass their criteria; the DR drill restores a snapshot < 30 minutes ago and the restored DB passes `audit_service.verify_chain()` with zero breaks; `DR_RUNBOOK.md` is signed off.
