## Phase 3: Entity Reporting, Lookup, Stats & Reputation (48 tests)

**Purpose:** Test the full entity reporting lifecycle end-to-end: report entity → lookup → stats → reputation → widget → compliance reports. Verify confirmation threshold, risk level computation, reputation scoring formula, public vs authenticated reputation, SVG widget, PII masking, and PDF/ZIP generation.

**Files involved:**
- `backend/app/api/v1/report.py` — POST /report, GET /lookup, GET /reports/stats, GET /reports/rbi/{quarter}, GET /reports/regulator/{quarter}
- `backend/app/api/v1/reputation.py` — GET /reputation/{vpa}, GET /reputation/{vpa}/public, GET /reputation/{vpa}/widget
- `backend/app/models/entity.py` — FlaggedEntity, EntityReport models
- `backend/app/services/intel/reputation_service.py` — compute_reputation, get_public_reputation, _tier, _recency_weight, _mask_entity
- `backend/app/services/compliance/rbi_report_builder.py` — RBIReportBuilder (reportlab PDF)
- `backend/app/services/compliance/export_pack.py` — generate_regulator_pack (CSV + manifest + ZIP)

**Key business logic (from code analysis):**
- Entity key = `{TYPE}:{value_lower}` (e.g., `PHONE:9876543210`)
- Confirmation threshold = 3 reports → `is_confirmed=1`
- Risk levels: `<3→low`, `3-4→medium`, `5-9→high`, `10+→critical`
- Reputation scoring formula: `base = min(50, log(1+count)*10) * recency_weight` + `ring_bump (15/25)` + `propagated*30` + `min(20, banks*5)`
- Reputation tiers: `clean (<20) | watch (20-49) | suspicious (50-79) | confirmed_scam (>=80)`
- Recency weights: `≤7d→1.0`, `≤30d→0.7`, `≤90d→0.4`, `≤180d→0.2`, `>180d→0.1`
- Public reputation: entity value masked (`xxx***xx`), count bucket: `none(0) | few(1-2) | several(3-10) | many(10+)`
- Widget color: green `≥80`, orange `≥50`, red `<50`, gray `unknown/0`
- RBI report uses reportlab, falls back to mock stats if DB query fails
- Export pack: entities CSV (PII masked) + sessions CSV + feedback CSV + audit CSV + signed manifest

## Bugs Found During Testing (Must Fix Before Continuing)

### Bug 1: Token Refresh Broken — Claim Name Mismatch
- **File:** `backend/app/api/v1/auth.py`, line 193
- **Code:** `family_id = decoded.get("family")`
- **Problem:** JWT claim is `family_id` (set in `jwt_service.py:62`), but refresh reads `family`
- **Fix:** Change to `family_id = decoded.get("family_id")`
- **Impact:** ALL token refresh attempts fail with 401 "Invalid token claims"

### Bug 2: Post-Logout Token Still Valid — Missing jti in Access Tokens
- **File:** `backend/app/services/auth/jwt_service.py`, `create_access_token()` (line 43-50)
- **Code:** `to_encode.update({"exp": expire, "type": "access", "ver": token_version})`
- **Problem:** No `jti` claim in access tokens. Logout (`auth.py:287`) checks `decoded.get("jti")` to revoke, but it's always None, so revocation never happens.
- **Fix:** Add `"jti": str(uuid.uuid4())` to the token claims
- **Impact:** After logout, access token remains valid for 30 minutes (until expiry)

### Execution Order After Fixes
1. Fix Bug 1 in `auth.py:193` — one character change
2. Fix Bug 2 in `jwt_service.py:49` — add `jti` claim
3. Restart backend
4. Re-run Phase 1 tests (should be 18/18 PASS)
5. Continue to Phase 2+
