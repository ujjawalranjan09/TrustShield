## Step 1: Merchant/VPA Reputation API

### 1a. Reputation scoring endpoint
**File:** `backend/app/api/v1/reputation.py` (new)

```
GET /api/v1/reputation/{vpa}
```

Score computation uses **existing data only** (no Neo4j dependency for v1):
- `report_factor` = `log(1 + report_count)` from `flagged_entities` table (entity_value matching the VPA)
- `cross_bank_factor` = count of `shared_entities` records for this VPA / 10 (capped at 1.0)
- `score = max(0, 100 - (report_factor * 50 + cross_bank_factor * 50))`
- `risk_level` = `critical` (score < 20), `high` (< 40), `medium` (< 60), `low` (>= 60)

If no data exists for the VPA, return `score: 100, risk_level: "unknown", report_count: 0`.

### 1b. Reputation widget
**File:** `backend/app/api/v1/reputation.py`

```
GET /api/v1/reputation/{vpa}/widget
```

Returns an **SVG badge** (not HTML — SVG is self-contained and embeddable anywhere):
- Green badge for score >= 80, yellow for >= 50, red for < 50
- Include `Cache-Control: public, max-age=3600` header

### 1c. Rate limiting
- Public (no auth): `10/minute` per IP — score + report_count only
- Authenticated (API key): `100/minute` — full details + scam_types + history

### 1d. Reputation history (authenticated)
```
GET /api/v1/reputation/{vpa}/history
```
Returns report timeline (dates, scam_types, reporter banks).

## Step 2: Consumer PWA + WhatsApp Bot

### 2a. PWA scanner endpoint
**File:** `backend/app/api/v1/consumer.py` (new)

```
POST /api/v1/consumer/scan
Body: {"text": "congratulations you won...", "language": "en"}
```

Lightweight endpoint for the consumer PWA:
- Runs through NLP pipeline (reuse `_get_services()` pattern from voice.py)
- Returns: `risk_score`, `risk_level`, `scam_type`, `warning_en`, `warning_hi`, `recovery_steps`
- No API key required (public endpoint, rate-limited: `20/minute` per IP)

### 2b. WhatsApp webhook
**File:** `backend/app/api/v1/whatsapp.py` (new)

```
POST /api/v1/whatsapp/webhook
```

Receives WhatsApp Business API webhook payloads:

- Reply with risk verdict + recovery steps via WhatsApp API
- Store scan event for analytics
- Config: `whatsapp_verify_token`, `whatsapp_access_token`, `whatsapp_phone_number_id` in config.py

Minimal PWA page:

- "Scan" button → calls `/api/v1/consumer/scan`
- Shows risk gauge + bilingual warning
- "What should I do?" accordion with recovery steps
- Installable as PWA (service worker + manifest)

## Step 4: Model Explainability Chat (RAG)

### 4a. Explainability chat endpoint
**File:** `backend/app/api/v1/explain.py` (extend existing)

Add to the existing explain router:
```
POST /api/v1/explain/chat
Body: {"question": "Why was session S-1234 flagged?", "session_id": "S-1234"}
```

### 4b. RAG pipeline (template-based, no LLM for v1)
**File:** `backend/app/services/explain/rag_chat.py` (new)

1. Look up session by `session_id` in `scan_events` table
2. Retrieve risk score, entities, classification, scam_type
3. Retrieve SHAP attributions from the risk scorer's explanation cache
4. Retrieve graph connections (if entity graph has data for this session)
5. Build structured answer from templates:
   - "Session was flagged because: [top contributing factors]"
   - "Detected entities: [entity list with descriptions]"
   - "Risk score: [score] ([level]) — [recommended action]"
6. Return answer + source references

### 4c. LLM upgrade (optional, env-gated)
**File:** `backend/app/services/explain/rag_chat.py`

If `settings.llm_api_key` is set, send the context + question to the LLM for a natural-language answer. Otherwise use template-based generation.

### 4d. Config
**File:** `backend/app/config.py`

Add: `llm_api_key: str = ""`, `llm_provider: str = "openrouter"` (values: `"openrouter"` | `"local"`)

## Step 5: Adaptive Bilingual Warnings

### 5a. Warning generator
**File:** `backend/app/services/nlp/warning_generator.py` (new)

Template-based warning generation:
- Input: `scam_type`, `risk_score`, `entities` (list), `locale` (en/hi)
- Templates organized by `scam_type × risk_level × locale`
- Entity-aware: "AnyDesk remote access was detected" not just "suspicious tool found"
- v1 supports EN + HI only (MR/TA/BN deferred to LLM integration)

### 5b. Wire into analyze/scan/voice responses
**Files:** `backend/app/api/v1/analyze.py`, `backend/app/api/v1/scan.py`, `backend/app/api/v1/voice.py`

Replace hardcoded warning strings with:
```python
from app.services.nlp.warning_generator import WarningGenerator
gen = WarningGenerator()
warning = gen.generate(scam_type=..., risk_score=..., entities=..., locale="en")
```

### 5c. LLM integration (optional, env-gated)
If `settings.llm_api_key` is set, use LLM to rewrite template warnings into natural-sounding regional language messages (MR/TA/BN). Fallback to template if LLM fails.

## Risks & Blockers

| Risk | Mitigation |
|------|------------|
| WhatsApp Business API approval takes time | Build webhook first, test with sandbox, apply for production access early |
| Consumer PWA needs its own domain/CORS config | Add PWA origin to `allowed_origins` in config |
| No `bank` role in User model | Add it in Step 7a — simple model change + migration |
| LLM costs for adaptive warnings | Default to templates; LLM only for regional languages (MR/TA/BN) |
| Reputation API cold-start on unknown VPAs | Return `score: 100, risk_level: "unknown"` — no error |
| Intervention endpoints are stubs (no real bank integration) | Document as v1; real integration requires bank API partnership |
| Regulator export may be slow for large quarters | Add background job + polling endpoint for large exports |
